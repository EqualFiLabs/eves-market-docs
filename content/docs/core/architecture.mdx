## System layers

The market protocol is exposed through a diamond. Facets share canonical market, book, fee, and resolution storage while keeping externally visible surfaces separated by job.

| Layer | Examples | Role |
| --- | --- | --- |
| Market core | `MarketFactoryFacet`, `MarketViewFacet`, `ParimutuelFacet` | Create and read orderbook and parimutuel markets. |
| CLOB | `CurveCLOBFacet`, spot book facets, trade routers | Post curves, fill liquidity, split or merge inventory, and operate ERC20 or ERC1155 books. |
| Resolution | OBR and settlement facets | Resolve markets, report CTF payouts, finalize pools, and redeem or claim. |
| Collateral | `EveUSD`, `SEveUSDVault`, lending routers | Wrap USDC, hold protocol collateral, distribute vault rewards, and borrow against vault shares. |
| Tokens | Gnosis CTF, `ParimutuelShareToken` | Represent outcome positions and pooled share balances. |

## Diamond model

Applications should call the diamond address for market functions. Individual facet contracts are implementation modules, not the integration endpoint.

The diamond owns protocol configuration such as default ConditionalTokens, collateral token, fee recipients, market duration bounds, fee splits, parimutuel settings, and creation permissions. These values are live governance or operator settings, so integrations should read `getMarketConfig` instead of hardcoding assumptions.

:::note Integration boundary
The diamond is the integration boundary. Use standalone contract addresses only for tokens, vaults, lending contracts, and auxiliary routers that are explicitly deployed outside the diamond.
:::

## Conditional tokens

Orderbook markets use Gnosis ConditionalTokens ERC1155 positions. The deploy path can deploy a standard Gnosis ConditionalTokens implementation when no address is provided, or it can wire an existing compatible CTF address into the diamond.

The protocol does not rely on a custom CTF token. Market creation prepares conditions lazily where possible, and combo books derive nested Gnosis CTF collection IDs from existing started markets.

## Standalone contracts

| Contract | Role |
| --- | --- |
| `EveUSD` | 18 decimal USDC backed collateral wrapper. |
| `SEveUSDVault` | ERC4626 style productive capital layer for eveUSD deposits and registered rewards. |
| `SEveUSDLending` | Borrowing surface backed by sEVEUSD shares. |
| `MakerLendingRouter` | Bundled borrow, market making, and unwind flows. |
| `ParimutuelShareToken` | ERC1155 share token minted and burned by the diamond for parimutuel markets. |
