## Facet inventory

The diamond exposes 52 facets. Each owns a narrow set of selectors and resolves against the diamond address. The notable splits to be aware of: the resolver registry is split across core, view, rewards, and reputation facets; parimutuel reads live on a separate view facet; market creation, market groups, and market views are distinct; and the curve and book CLOB surface is split into inventory, lifecycle, trade, and view facets each.

| Group | Facets |
| --- | --- |
| Core | DiamondCut, DiamondLoupe, Ownership, FeeConfig |
| Markets | MarketFactory, MarketGroup, MarketView, MarketSettlement |
| Curve CLOB | CurveInventory, CurveLifecycle, CurveCLOB, CurveView |
| Book CLOB | Book, BookOrder, BookTrade, BookView |
| Multi-outcome and combo | MultiOutcomeOrderbook, MultiOutcomeOrderbookView, NativeBinaryPosition, ComboCore, ComboSettlement, ComboBranch, ComboView, ComboMarket |
| Parimutuel | Parimutuel, ParimutuelView |
| Parlay | ParlayAdmin, ParlayUnderwriting, ParlayBudget, ParlaySettlement, ParlayBook, ParlayView, ParlayMulticall |
| Resolution | OBRResolution, ResolverRegistry, ResolverRegistryView, ResolverRegistryRewards, ResolverRegistryReputation, ResolverJury |
| Margin and MLO | MarginAccount, MarkOracle, QuoteEnvelope, MLOPredictionAdapter, MLOPredictionCurve, MLOPredictionTrade, MLOPredictionSettlement |
| Routing | TradeRouter, TradeRouterSell, DelayedOrder, FeeRouter, BondManager, BondTokenGate |

## Creation surfaces

The canonical creation entry points take typed param structs carrying display and external reference metadata. Legacy positional overloads exist for backwards compatibility but omit that metadata.

```solidity createMarket
createMarket(MarketCreationParams params)
createMarketWithCollateralProfile(uint8 profileId, MarketCreationParams params)
createMarkets(MarketCreationParams[] params)
```

`createMarketGroup` and `createMarketGroupFromExisting` batch related markets under one group id. `createMultiOutcomeMarket` and its profile variant create multi-outcome orderbook markets with one outcome book per outcome.

```solidity createParimutuelMarket
createParimutuelMarket(CreateParimutuelMarketParams params)
createParimutuelMarketWithCollateralProfile(uint8 profileId, CreateParimutuelMarketParams params)
```

Combo markets are created with `createComboMarket`, not a generic book creator.

## Fee surfaces

Per-surface fee splits are read through the fee config facet and the market view fee config views. Each surface exposes its take rate plus the maker, creator, protocol, seniorPool, resolver, and etRisk legs as applicable. Builders computing splits must include all legs; omitting resolver or etRisk will mis-state net proceeds.

## Execution rules

Taker fills should pass expected generations or expected commitments. Frontends should preview output, enforce slippage, avoid stale curve fills, and distinguish book fills from market-side convenience routes.

The trade router facets wrap the fill path so a user can act from a USDC or eveUSDC balance directly: `buyWithUSDC`, `buyWithEveUSDC`, `splitWithUSDC`, `buyWithCollateral`, and the matching sell variants.

For scheduled markets, user-facing actions should check `tradingStartTime` and `expiryTime` before enabling submit buttons. For market-side CLOB books, contract execution is open only after the market has started and before expiry while the market is in the executable trading state.

## Bond and resolution config

`getMarketConfig` exposes `bondToken`, `marketCreationBond`, `resolutionBondL1`, and `resolutionBondL2`. The bond token defaults to eveUSDC. Builders should show the configured bond token symbol and decimals by reading the ERC20 metadata.

The OBR resolution ABI exposes `settleMarket`, `settleMarketEarly`, `openResolution`, `disputeResolution`, `finalizeResolution`, `adminFinalizeResolution`, and `finalizeFromJury`, plus `resolutionMode()`. The mode gates which path is live: under `CreatorAdminBootstrap` the owner can finalize any market, while under `ObrJury` disputes escalate to the resolver jury.
