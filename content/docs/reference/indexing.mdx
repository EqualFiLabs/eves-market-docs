## Market discovery

Index `MarketCreated` for market discovery, then enrich from `getMarketMetadata`, `getMarketInfo`, and `getMarketTokenInfo`.

The event gives core identity data, but read functions provide current state, fee snapshots, trading start, position token type, and token IDs.

## Lazy books

Market creation gives deterministic YES and NO book IDs. It does not guarantee that a `BookCreated` event has been emitted for those side books.

Indexers should:

- Store deterministic side book IDs from `getMarketSideBook`.
- Treat `isBookMaterialized(bookId) == false` as a valid pre-liquidity state.
- Add full book records when `BookCreated` fires or when `getBookInfo` succeeds.
- Track combo, native, and ERC20 spot books from explicit `BookCreated` events.

## Activity streams

| Event family | Use |
| --- | --- |
| Market and group creation | Discovery, market groups, and upcoming calendars. |
| Book creation | Materialized books, ERC20 spot books, combo and native books. |
| Curve posted or updated | Maker liquidity state. |
| Curve filled or trade routed | Taker execution and volume. |
| Delayed order submitted, processed, expired | FIFO protected execution lifecycle. |
| Parimutuel shares bought | Pool entry, implied probability, and share balances. |
| Resolution finalized | Settlement cutover. |
| Payout claimed or redeemed | Post-resolution user accounting. |

## Resolution finality

Resolver jury events are the primary signal for dispute finality and should be indexed as a first-class stream. The family covers dispute initiation, randomness commit and reveal, committee selection, vote commit and reveal, tally, appeal, and finalization, plus the resolver epoch rotation events that mark when membership changes.

Resolver reward accrual and finalization events mark when trading-fee rewards become claimable per epoch. Before an active epoch exists, resolver and identity fees route to the treasury, so indexers tracking protocol revenue should treat treasury receipts as authoritative during bootstrap.

## Fee distribution

Every fee leg resolves in the transaction that produced it, so a fee event is final rather than pending a later claim. Indexers tracking protocol revenue must capture all sinks, not just treasury: maker and creator accruals on the fee router, the seniorPool leg into SeniorCapitalPool, the resolver leg accrued per epoch, and the etRisk leg pushed to EtRiskStakingRewards. The seniorPool and etRisk legs fall back to treasury when their recipient is ineligible, so a treasury receipt does not always mean the protocol fee alone.

EtRisk staking events cover reward notification, staking and unstaking, claims, and the treasury fallback when a series is inactive or has no stakers.

## Combo indexing

Combo and native books should store their leg list offchain from the transaction input or app metadata. The onchain book tells you the base token and position ID, but human-readable combo UX needs the leg markets and index sets. Native binary and combo positions are minted by the Eves-native position manager, so track position metadata through the combo view facet rather than assuming Gnosis CTF wallets for every leg.
