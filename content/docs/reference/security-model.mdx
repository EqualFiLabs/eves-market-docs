## Collateralization

Orderbook CTF positions are backed by the market collateral locked in ConditionalTokens. Parimutuel payouts are limited to the finalized payout pool. CLOB maker curves are collateralized by escrowed base inventory or quote collateral.

There is no automatic liquidity guarantee. If no maker has posted collateralized liquidity, a book may exist by ID but still be empty.

## Admin powers

The current alpha is not governance-free. Operators can configure core addresses, collateral profiles, fee splits, duration bounds, permissionless creation, creation fees, parimutuel settings, resolution windows, bond token and bond amounts, resolver jury settings, the senior capital pool, the EvRisk staking rewards contract, and ConditionalTokens references.

The most consequential admin lever is `resolutionMode`. Under `CreatorAdminBootstrap` the owner can `adminFinalizeResolution` any market and report an outcome unilaterally, so that mode carries a direct trust assumption. It is intended for early launch and should be migrated to `ObrJury` before production use, after which disputes escalate to the resolver jury and the admin finalize path is unavailable.

Integrators should surface this as an actively governed alpha protocol and expose the live resolution mode.

## Fee routing

Fee splits fan out to several recipients at trade time. The seniorPool leg lands in SeniorCapitalPool, the resolver leg accrues to compliant resolvers per epoch, and the evRisk leg is pushed to EvRiskStakingRewards for active stakers. Each non-treasury leg has a treasury fallback when its recipient is ineligible, so no fee gets stranded, but a security review must trace every leg to its destination rather than assuming all revenue reaches the treasury.

Identity mint and resolver epoch candidate fees route to the treasury. The bond token defaults to eveUSDC, so bond slashing and resolution bonds are ERC20 denominated, never native ETH or `msg.value`.

## CTF assumptions

The protocol targets Gnosis ConditionalTokens compatibility for its orderbook CTF surface, while native binary and combo positions use an Eves-native ERC1155 position manager. Combo behavior that relies on CTF uses standard collection ID and position ID algebra. Wallet metadata support may vary because the protocol metadata helper lives on the diamond, not on every raw CTF wallet integration.

## Integration safety

Applications should preview routes, enforce slippage, check market timing, pass expected curve generations or commitments, avoid stale fills, and treat lazy books as a normal state rather than an indexer error.

For profile markets, applications should also display the collateral token and payout unit at every value-moving step. Treat cross-collateral portfolio totals as offchain estimates, not protocol balances.
