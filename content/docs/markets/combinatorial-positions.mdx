## The native position stack

The diamond carries a full native position stack: native binary conditions, combo core operations, branch manipulation, settlement, views, and combo market creation. These build on an Eves-native ERC1155 position manager rather than requiring Gnosis CTF for every leg.

| Facet | Owns |
| --- | --- |
| NativeBinaryPositionFacet | prepare, split, merge, redeem native binary conditions |
| ComboCoreFacet | prepare combo conditions, split, merge, wrap, unwrap combos |
| ComboBranchFacet | split/merge on condition, extract/inject NO legs, convert NO baskets |
| ComboSettlementFacet | compress, redeem, payout |
| ComboViewFacet | native metadata, compressibility, compression preview |
| ComboMarketFacet | create combo markets, compute book ids |

## Native binary positions

`NativeBinaryPositionFacet` lets a market prepare a binary condition and split collateral into native YES and NO positions, merge them back, and redeem after resolution. This is the native equivalent of the CTF split and merge flow, without depending on the external ConditionalTokens contract.

## Combo positions

A combo position is an ERC1155 scenario token that represents a conjunction across multiple markets. A trader can express `A AND B AND C` as one position ID instead of managing separate branches.

These positions are parlay-like, but they are decomposable conditional positions: parts of the tree can be split, merged, compressed, wrapped, or traded when the holder owns or acquires the required complementary branches.

`ComboCoreFacet` prepares the combo condition and exposes split, merge, wrap, and unwrap. Wrapping bundles a combo into a single transferable token; unwrapping decomposes it back into its legs.

## Combo markets

Combo markets are created with `createComboMarket`, not a generic book creator. They use native binary or combo ERC1155 positions as base inventory and belong to the combinatorial flow, not the ERC20 spot surface. `computeComboBookId` derives the deterministic book id for a combo market.

## Branch manipulation

Moving around the combo tree is explicit. `ComboBranchFacet` exposes the operations that restructure a position when the complementary branches are available.

| Function | Effect |
| --- | --- |
| `splitComboOnCondition` | Split a combo further on a new condition. |
| `mergeComboOnCondition` | Merge a split back up the tree. |
| `extractComboNoLeg` | Pull a NO leg out of a combo. |
| `injectComboNoLeg` | Add a NO leg back in. |
| `convertComboNoToYesBasket` | Convert a NO leg into an equivalent YES basket. |
| `mergeComboNoFromYesBasket` | Reassemble a NO leg from a YES basket. |

:::warning Complement requirement
`NO(A AND B)` is not the same as `NO(A) AND NO(B)`. Combo UX must distinguish the complement of an entire scenario from individually negated legs. Extraction and conversion only work when the holder owns the complementary branches needed to merge.
:::

## Parlays

The parlay facet set is the underwriter-backed realization of multi-leg scenario tickets. A parlay is expressed as an ERC1155 ticket whose payout depends on every leg resolving correctly, with payout tiers and an underwriting fee.

| Facet | Owns |
| --- | --- |
| ParlayUnderwritingFacet | post and fill parlay offers and requests |
| ParlayBudgetFacet | funded budgets that sponsor offer and request batches |
| ParlaySettlementFacet | finalize ticket buckets and claim tickets |
| ParlayBookFacet | create parlay ticket books |
| ParlayViewFacet | template and ticket ids, offer/request state, ticket URIs |
| ParlayAdminFacet | parlay config and strategy emission |
| ParlayMulticallFacet | batched calls |

Parlay tickets are distinct ERC1155 balances minted by `ParlayTicketToken`. They are not the same as combo positions: a combo decomposes into tradeable legs, while a parlay ticket settles as one unit against its leg set.

## Pricing

A core combo claim pays at most one collateral unit. An 8-leg combo is not worth more than one collateral unit because it has 8 legs; the upside comes from buying a low-probability scenario below its maximum payout.

Makers can charge a premium above implied fair probability, but the book should not accept prices above the maximum redeemable payout for the claim.
